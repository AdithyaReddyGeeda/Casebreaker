5. ROOT CAUSE (Proven by Runtime Evidence)
────────────────────────────────────────────────────────────────────────────────

The Tripo v3.0 text_to_model task, as originally submitted, produced a
single-mesh rigid GLB with no facial blendshapes, no skeleton, no jaw bone,
no head bone. The lip-sync engine had nothing to bind to on this asset.

Brian.glb (the non-Tripo fallback model) has a skeleton but no blendshapes
and no jaw bone, so it also cannot produce true lip motion - only the
procedural body/head bob fallback.

   - Replaced the single model_version fallback with a progressive retry

         2. drop workflow
         3. drop face_rig + workflow
         4. drop model_version only
         5. minimal (no version/face_rig/workflow)
     This ensures we try the most capable rigged request first and only
     degrade when the API forces it.

7. WHY THE FIX ISN'T VISIBLE YET
────────────────────────────────────────────────────────────────────────────────

The API route (app/api/tripo-model/route.ts) short-circuits to the cached
public/models/dr_fenn_tripo.glb whenever it exists. The "Load/Generate
Dr. Fenn" button hits the API without ?regenerate=1, so it keeps serving
the OLD rigid GLB. Every N1 nuclear dump has shown the identical node UUID
(tripo_node_bc9d3e02-c774-417c-984b-62c237d48058), proving the cache was
never invalidated.

The new face_rig/workflow params in tripoService.ts are real and wired,
but they only take effect when a true regenerate is triggered. Options:
  1. Delete character-preview/public/models/dr_fenn_tripo.glb manually,
     then click Load/Generate.
  2. Run `npm run seed:dr-fenn` from character-preview/ (it passes
     ?regenerate=1 and re-downloads the fresh GLB).
  3. Wire a "Force Regenerate" button (in progress - generateTripoModel
     already accepts a forceRegenerate arg, just needs a UI trigger).
